# cskburn 烧录问题排查

烧录失败时，cskburn 会输出形如 `ERROR [E3001]: Serial port not found` 的信息。方括号内的错误码对应唯一的根因，按下文对照处理即可。

---

## 常见场景

### 无法连接设备

出现 `E3xxx` 或 `E4xxx` 时：

- 确认 USB 线已接好，且为数据线而非充电线。
- 按住 BOOT 按键（或下载键）的同时复位芯片，确保进入下载模式。
- 检查 TX/RX 是否交叉、GND 是否接通、DTR/RTS 是否连接——DTR/RTS 用于自动复位和拉 BOOT，缺一不可。
- 确认 `-C` 选择的芯片系列与实际芯片一致：CSK6 用 `-C venus`，LS26 用 `-C arcs`。芯片系列错误会导致握手协议不匹配。
- 少数板子使用非常规的复位电路，可用 `--reset-strategy <name>` 显式指定：
  - `dtr-boot`：DTR→BOOT、RTS→RESET（LS26 ARCS-MINI）
  - `rts-boot`：RTS→BOOT、DTR→RESET（CSK4/CSK6 默认）
  - `rts-boot-inv`：同 `rts-boot` 但 BOOT 高电平有效（等价于旧的 `--update-high`）
  - `dual-npn`：差分驱动的 NPN 三极管对（LS26 ARCS-EVB）
  - 默认 `auto` 会按芯片自动选择，LS26 下会在 `dtr-boot` 和 `dual-npn` 之间交替重试。

### 烧录中途失败

出现 `E5xxx` 或 `E7xxx` 时：

- **优先降低波特率**：先试 `-b 1500000`，仍失败则进一步降至 `-b 921600`。高速烧录对 USB 转串口芯片和线材质量较敏感。
- 更换质量较好的 USB 数据线，并直连主板 USB 口，避开 Hub。
- 确认开发板供电充足，尤其是外接模组的场景。

### 校验未通过

开启 `--verify-all` 后看到 `E8003: Verification failed`：

- 先重烧一次，排除偶发因素。
- 若稳定复现，通常是 flash 坏块、擦除不彻底或电源噪声引起写入位翻转，需从硬件层面排查。

## 错误码速查

### E1xxx — 参数不合法

| 错误码 | 含义 |
| --- | --- |
| `E1001` | 参数格式或取值不合法，文案末尾指出具体参数 |
| `E1002` | 未指定烧录端口，使用 `-s <port>` 或 `-u <bus>:<dev>` |
| `E1003` | 操作与当前模式/芯片不兼容（如 NAND 仅 CSK6 支持、RAM 模式不支持 `--verify`） |
| `E1004` | 单次 `--read` / `--erase` / `--verify` 超过 20 个分区上限 |
| `E1005` | 地址或长度未对齐（Flash 需 4K 对齐，NAND 需 512 字节对齐） |
| `E1006` | 地址或长度超出目标容量 |

### E2xxx — 本地文件 I/O

| 错误码 | 含义 | 排查 |
| --- | --- | --- |
| `E2001` | 读取输入文件失败 | 检查路径、文件存在性、读权限 |
| `E2002` | 写入输出文件失败（`--read`） | 检查目录存在性、磁盘空间、写权限 |
| `E2003` | HEX 文件解析失败 | 更换正确的 HEX 文件，或转换为二进制 |

### E3xxx — 串口打开/配置

| 错误码 | 含义 | 排查 |
| --- | --- | --- |
| `E3001` | 串口设备不存在 | 检查设备已插入、驱动已安装、端口名正确（Linux `/dev/ttyUSB0`、macOS `/dev/cu.usbserial-xxxx`、Windows `COMxx`） |
| `E3002` | 无访问权限 | Linux 下将用户加入 `dialout` 组：`sudo usermod -aG dialout $USER`，重新登录生效 |
| `E3003` | 串口被占用 | 关闭占用端口的串口终端、IDE 监视器或其他 cskburn 实例 |
| `E3004` | 打开串口失败（其他） | 更换 USB 口或数据线后重试 |
| `E3005` | 串口参数配置失败 | 更换线材或重装 USB 转串口驱动 |
| `E3006` | 串口驱动不支持目标波特率 | 降低 `-b`；或更换上限更高的 USB 转串口芯片 |

### E4xxx — 探测/复位

| 错误码 | 含义 | 排查 |
| --- | --- | --- |
| `E4001` | 复位后芯片未响应 | 最常见。按优先级检查：BOOT 引脚、TX/RX 接线、DTR/RTS 接线、`-C` 芯片系列、供电；特殊复位电路的板卡需用 `--reset-strategy <name>` 显式指定（参见下方）；`-b` 过高触发驱动静默降档也会落到这里，可尝试降低 `-b` |
| `E4002` | 控制 RTS/DTR 失败 | USB 转串口设备异常，重新插拔或更换 |

### E5xxx — 进入更新模式

这一阶段按协议子步骤细分，不同错误码对应不同子阶段，可直接定位故障点：

| 错误码 | 含义 | 排查 |
| --- | --- | --- |
| `E5001` / `E5005` | 设备侧（ROM / burner）拒绝切换波特率 | 降低 `-b` |
| `E5002` / `E5006` | 切换波特率后同步丢失 | 降低 `-b`；更换线材 |
| `E5003` | 向 RAM 加载 burner 失败 | 降低 `-b`；自定义 `--burner` 时确认与芯片匹配 |
| `E5004` | burner 加载完成后未启动 | `-C` 与芯片不一致；或自定义 `--burner` 不兼容 |

### E6xxx — 设备/Flash 信息

| 错误码 | 含义 | 排查 |
| --- | --- | --- |
| `E6001` | 读取芯片 ID 失败 | 通常伴随连接不稳，先处理 E4/E5 |
| `E6002` | 读取 Flash ID 失败 | 同上 |
| `E6003` | 未检测到 Flash（ID 全 0 或全 FF） | Flash 焊接不良、型号不支持或未供电 |
| `E6004` | NAND 初始化失败 | 确认 `--nand-cmd` / `--nand-clk` / `--nand-dat0..3` 与硬件匹配，以及 `--nand-4bit` 模式正确 |

### E7xxx — Flash / NAND / RAM 读写擦除

| 错误码 | 含义 | 排查 |
| --- | --- | --- |
| `E7001` | 读取 Flash 失败 | 降低 `-b`、检查线材 |
| `E7002` | 擦除 Flash 失败 | Flash 写保护位、地址越界，或大块擦除需增大 `--timeout` |
| `E7003` | 写入 Flash 失败 | 降低 `-b`、检查供电稳定性 |
| `E7004` | 写入 NAND 失败 | 同 E7003 |
| `E7005` | 写入 RAM 失败（`-r` 模式） | 确认 `--jump` 地址与 RAM 区间合法 |

### E8xxx — 校验

| 错误码 | 含义 | 排查 |
| --- | --- | --- |
| `E8001` | 设备端 MD5 计算失败 | 伴随通信异常，参考 E7xxx |
| `E8002` | 本地 MD5 计算失败 | 极少见，通常是系统异常 |
| `E8003` | 校验失败（MD5 不匹配） | 先重烧；稳定复现多为 flash 坏块或电源问题 |

### E9xxx — USB 烧录（仅限 `-u` 模式）

| 错误码 | 含义 | 排查 |
| --- | --- | --- |
| `E9001` | USB 后端初始化失败 | Linux 检查 libusb 运行时；Windows 确认已安装 WinUSB 驱动 |
| `E9002` | 未找到 USB 设备 | 确认设备已进入 USB 下载模式；必要时用 `--usb <bus>:<dev>` 精确指定 |
| `E9003` | 进入 USB 更新模式失败 | 设备端 bootrom 握手失败，重新上电重试 |
| `E9004` | USB 数据写入失败 | 换 USB 口/线缆重试；若稳定复现上报维护者 |

### D## — 设备侧状态字节

形如 `ERROR [D0A]: Device reported error: partition 1`，前缀为 `D` 表示芯片内 burner 返回的业务状态码：

- `D0A`：Flash 繁忙或写入队列满。工具会自动延迟重试；若仍复现，通常是擦除异常或供电不稳。
- 其他字节码含义依具体命令而定，请结合上下文（如 `partition N`）反馈给维护者。

## 推荐排查顺序

1. **单独验证连接**：`cskburn -C <family> -s <port> --chip-id`。成功读到芯片 ID 即说明连接正常，问题在后续阶段。
2. **使用默认参数**：`cskburn -C <family> -s <port> 0x0 <file>`，观察具体错误码。
3. **降低波特率**：`-b 1500000` 或更低。
4. **增大容错**：`--probe-timeout 20000 --reset-attempts 8`。
5. **开启日志**：`-v` 输出调试信息，`--trace` 输出协议报文。
6. **事后校验**：`--verify-all` 或 `--verify addr:size` 抽检已写入内容。
